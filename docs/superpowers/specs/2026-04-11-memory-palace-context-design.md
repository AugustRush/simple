# Memory Palace Context Design

**Goal:** Rebuild context memory around fixed palace loci, typed memory items, and a SQLite-backed retrieval layer while preserving human-readable memory views.

## Problem

The current system splits memory into:

- `MemoryPalace`: markdown files grouped by legacy chapters
- `ContextManager` / `LTMStore`: JSON fact store with model-generated categories

This creates three structural failures:

1. Memory placement is unstable because categories are generated by the model.
2. Retrieval is flat BM25 over free-form facts instead of routed by memory type and scope.
3. Human-readable memory and machine-optimized memory are not the same system.

## Design Principles

1. Palace loci are fixed and stable.
2. New memory is routed, not categorized ad hoc.
3. Machine storage is the source of truth.
4. Human-readable markdown is a projection of structured memory.
5. Current-session recall and long-term recall are different layers.

## Palace Loci

Top-level loci are fixed:

- `identity`
- `projects`
- `people`
- `concepts`
- `episodes`
- `tasks`
- `procedures`
- `archive`

Each locus may have stable secondary anchors:

- `identity/user`
- `projects/<project_slug>`
- `people/<person_slug>`
- `concepts/<concept_slug>`
- `episodes/<session_id>`
- `tasks/<task_id>`
- `procedures/<procedure_slug>`
- `archive/<bucket>`

## Memory Layers

### Working Memory

- Source: `ctx.messages`, current tool results
- Scope: current inference only
- Storage: in-memory only

### Episodic Memory

- Source: session staging + consolidation
- Scope: one session / one event
- Locus: `episodes/<session_id>`

### Semantic Memory

- Source: extracted stable facts
- Scope: long-lived
- Loci: `identity`, `projects`, `people`, `concepts`

### Procedural / Task Memory

- Source: extracted next steps, workflows, commitments
- Scope: ongoing
- Loci: `tasks`, `procedures`

## Source of Truth

SQLite becomes the primary structured memory store.

Core record fields:

- `id`
- `locus`
- `entity`
- `memory_type`
- `content`
- `importance`
- `status`
- `source_session`
- `confidence`
- `created_at`
- `updated_at`

SQLite responsibilities:

- transactional writes
- indexed retrieval
- FTS-backed content search when available
- grouping by locus/entity/status

## Markdown Projection

Markdown remains available as a projection layer under `~/.agent/memory`.

Projection rules:

- one stable file per locus/entity anchor
- content rendered from active structured memory items
- projection is regenerated after writes
- legacy chapter names map to the new loci for compatibility

Initial compatibility mapping:

- `knowledge` -> `concepts`
- `archive` -> `archive`
- `people` -> `people`
- `projects` -> `projects`
- `tasks` -> `tasks`

## Routing Rules

Memory extraction must route to a fixed locus:

- user preferences / style / constraints -> `identity`
- project background / decisions / risks -> `projects`
- person-specific facts -> `people`
- stable concepts / definitions -> `concepts`
- session summary / “what just happened” -> `episodes`
- commitments / open loops -> `tasks`
- reusable workflows / patterns -> `procedures`
- superseded or stale items -> `archive`

## Retrieval Strategy

Two retrieval paths exist:

1. Implicit retrieval for prompt injection
   - use long-term memory only
   - route by heuristics to likely loci
   - query SQLite search with locus filters

2. Explicit context retrieval
   - include current session staging first
   - then include routed long-term memory

Priority by query intent:

- recent conversation -> `episodes` + current staging
- user preference -> `identity`
- active work -> `projects` + `tasks`
- “how do we usually do this” -> `procedures`
- concept lookup -> `concepts`

## Consolidation

Sleep/consolidation no longer creates arbitrary categories.

It must:

1. create one episodic session summary item
2. extract stable semantic / procedural / task items using fixed loci
3. upsert or append items into SQLite
4. archive or supersede stale items when necessary
5. clear the per-session staging buffer

## Migration

Migration keeps the external agent behavior working while replacing internals:

1. Introduce SQLite-backed store behind the current context manager.
2. Keep compatibility wrappers for `LTMStore` and legacy categories.
3. Update retrieval and consolidation to target fixed loci.
4. Expand `MemoryPalace` to expose the new loci and aliases.

## Risks

- FTS support may vary by SQLite build, so fallback search is required.
- Markdown projection can drift if manual file edits are treated as authoritative. In this phase, SQLite remains the source of truth.
- Single-file architecture makes the refactor mechanically risky; tests must lead the change.
